## SOP Phase Summary

### Phase 1: Pre-Execution Discovery
Collect and display all runtime parameters before executing anything:
- Primary theme name, project path, docs namespace, screenshot location
- **Existing Site Audit**: Ask the user explicitly — *"Is there an existing version of this website running locally?"* If yes, record its path. A yes answer makes Phase 3 Step 2 **mandatory**.
- Auto-detect container runtime (scan `.ddev/`, `.lando.yml`) — never hardcode
- Run `git status` — block on dirty working tree
- Display parameters as a table, ask for user approval before proceeding

### Phase 2: Establish the Baseline Backup
- Clone stable theme to timestamped working directory
- Refactor machine names in config files ONLY (`.info.yml`, `.breakpoints.yml`,
  `.theme`, `.libraries.yml`) — never touch binary assets or CSS class names
- Activate via `[runtime_wrapper] drush`
- Commit with explicit path only — never `git add .`

### Phase 3: Screenshot Ingestion & Component Mapping
- Save screenshots to `designs/` inside the new theme
- **Legacy Audit (mandatory if existing site confirmed in Phase 1 — do not skip)**:
  Detect legacy site runtime independently, then audit: content types, menus,
  block placements, taxonomy vocabularies, active layout modules.
  Write findings to `audits/legacy_dissection.md` inside the new theme.
  This output directly drives template suggestion hooks and sidebar wiring.
- Decompose screenshot into horizontal bands
- Cross-reference against user-specified base theme component library
- Identify gaps needing bespoke SDCs
- Draft `theme_component_mapping_plan.md` into the user's docs folder
- Commit all artifacts with explicit paths — never `git add .`
- Hard stop for user approval before proceeding to Phase 4

### Phase 4: Page Template Architecture
- Inventory all page types in scope (Canvas full-width, sidebar docs, utility)
- Identify the homepage node; do not create one if it already exists
- Author `page--front.html.twig`: suppress page title, full-width `{{ page.content }}`
- Author sidebar documentation variant: `sidebar_first` region for TOC/index menu
- Wire up any new regions in `[theme].info.yml`
- Commit all templates and region additions with explicit paths
- Approval checkpoint before Phase 5

### Phase 5: Implementation Execution
- Author markup as SDC Twig templates — no loose HTML files
- Global CSS tweaks for existing native components → `css/base.css`
- Bespoke component CSS → encapsulated inside `components/[name]/[name].css`
  — NEVER in `base.css`
- Autonomous content via Heredoc stdin (no PHP files written to disk):
  ```bash
  cat << 'EOF' | [runtime_wrapper] drush scr -
  <?php
  // ... \Drupal::service('ai_agents') invocation ...
  EOF
  ```
- Commit SDC bundles and CSS wrappers

### Phase 6: Verification
- Manual hold: instruct user to `drush cr`, assemble in Canvas UI, provide URL
- Load URL in browser, visual regression against original screenshot
- Verify CSS cascade encapsulation
- **Failure path**: regression fails → `git revert HEAD` → return to Phase 5
  with failures documented as constraints

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Framework-agnostic master SOP | Must work for any base theme, not just Dripyard |
| Runtime auto-detected, never hardcoded | Prevents crashes on Lando, bare metal, etc. |
| Heredoc stdin for PHP payloads | `/tmp` not mounted in DDEV containers; zero disk pollution |
| SDC CSS always encapsulated locally | Prevents global cascade pollution in Canvas architecture |
| Git snapshot after every phase | Enables clean `git revert HEAD` for any single phase |
| Plan committed before approval halt | Prevents mapping plan floating un-tracked between conversation turns |
| `git add` always path-scoped | Prevents accidental staging of unrelated files |
